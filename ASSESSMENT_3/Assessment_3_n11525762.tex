\subsection{Preliminary Design – Web Interface}

The PowerStake web interface acts as the primary interaction point between the user and the system. It is responsible for presenting power monitoring data, AI-generated recommendations, and system alerts in a clear and accessible format. The interface supports the validation plans developed in Assessment 2 by allowing users to view system outputs and take action based on those outputs.

The interface is implemented using a lightweight web-based dashboard approach, with Streamlit selected for the prototype. This decision was made due to its rapid development capability, direct integration with Python-based AI logic, and suitability for low TRL (TRL 3–4) systems where functionality is prioritised over production-level design.

The core features of the interface include:
\begin{itemize}
    \item Real-time display of power usage data
    \item Alerts for abnormal appliance behaviour
    \item AI-generated energy-saving recommendations
    \item Device-level monitoring overview
    \item Manual override controls for switching actions
\end{itemize}

The system architecture for the interface follows a simple data flow:

\begin{center}
Power Monitoring → AI Subsystem → Web Interface → User Decision → System Control
\end{center}

This ensures that data collected from the hardware and processed by the AI subsystem is directly translated into actionable outputs for the user.

From a feasibility perspective, the web interface is highly achievable within the project timeframe. The use of Streamlit removes the need for complex frontend/backend separation, allowing the interface to be developed alongside the AI subsystem. However, limitations include reduced scalability and limited customisation compared to full-stack web applications. These limitations are acceptable for a prototype-level implementation.

The PowerStake web interface prototype is available at:

\begin{center}
\texttt{https://egh419assessment3.streamlit.app/}
\end{center}

--

A review of existing products such as TP-Link smart plugs and Belkin energy monitoring devices indicates that most current solutions focus on hardware control and data visualisation, with limited protection around intelligent energy decision-making systems.

The key IP asset identified for PowerStake is the AI-based decision engine that converts raw power data into actionable recommendations. This includes logic related to detecting standby usage patterns, identifying abnormal behaviour, and generating automated suggestions.

This asset is best protected through trade secrets and software copyright. Trade secrets are appropriate as the algorithmic logic can be kept confidential and is difficult to reverse engineer without access to the internal system. Copyright applies to the implementation of the software code and interface design.

\subsection{Risk Assessment}
o	All team members describe 2 risks different from the other members of the team.
o	Identify one (1) industry standards or regulations that might impact your product.
o	Identify one (1) risks related to the operational, technical, financial or time feasibility of your product and undertake the risk assessment and management for them. Refer to tutorials in Week 7 and Week 9.
--


Two key risks associated with the web interface subsystem are identified.

The first risk is usability risk. If the interface is too complex or unclear, users may ignore recommendations or fail to act on alerts. This directly reduces the effectiveness of the system. To mitigate this, the interface design prioritises simplicity, using clear visual indicators and minimal user interaction requirements.

The second risk is data accuracy perception. If users perceive the displayed data or recommendations as incorrect, trust in the system will decrease. This risk is mitigated by clearly presenting data sources and ensuring consistency between displayed values and system outputs.

A relevant industry consideration is data privacy and security. The system must ensure that user energy data is handled securely, particularly when transmitted over Wi-Fi networks. This aligns with general data protection practices and consumer device standards.

A key feasibility risk is technical integration between the AI subsystem and the interface. Delays or errors in data transmission could result in outdated or incorrect recommendations. This is managed by using a simplified architecture and limiting dependencies between components during the prototype phase.
